======================================================================
     AGENTIC KNOWLEDGE SUPPORT SYSTEM (MultiDoc2Dial) - SUMMARY
======================================================================

1. MODELS USED
----------------------------------------------------------------------
*   Retriever (Embeddings): `sentence-transformers/all-MiniLM-L6-v2` 
    *   Role: Encodes queries and Knowledge Base documents into 384-dimensional dense vectors for rapid FAISS similarity search.
*   Reranker (Cross-Encoder): `cross-encoder/ms-marco-MiniLM-L-6-v2`
    *   Role: Computes high-accuracy semantic relevance scores between the query and the retrieved chunks to eliminate hallucinations.
*   Generator LLM: `google/flan-t5-base`
    *   Role: Generates the final natural language answer using Parameter-Efficient Fine-Tuning (PEFT/LoRA) for alignment.
*   Tool Policy Classifier: `distilbert-base-uncased`
    *   Role: Fine-tuned as a 3-class router (ANSWER, TICKET, REJECT) to autonomously dictate agent tool usage.

2. PIPELINE ARCHITECTURE (INFERENCE FLOW)
----------------------------------------------------------------------
The system utilizes an optimized 2-Phase architectural flow inside `generator.py`:

*   [PHASE 1] Zero-Shot Keyword Triage: 
    The query is scanned against a hardcoded list of domain keywords. If no keywords are found, it instantly returns the `Reject` tool response, drastically reducing computational overhead.
*   [PHASE 2] Deep Semantic Triage: 
    If keywords are present, `distilbert` computes class probabilities to determine if the query can be automatically answered (SearchKB) or requires human escalation (CreateTicket).
*   [PHASE 3] Retrieval & Reranking: 
    The `SearchKB` tool performs a FAISS semantic search, followed by a Cross-Encoder rescoring step. If the highest reranker score is extremely poor (<-2.0), the system automatically triggers the Fallback Escalation Tool.
*   [PHASE 4] Grounded Generation: 
    The FLAN-T5 model receives a highly constrained prompt containing the top-3 reranked contexts. The model generates a concise response with mandatory inline citations (e.g., [1]).

3. TOOL POLICY TRAINING HYPERPARAMETERS
----------------------------------------------------------------------
*   Hardware Engine: PyTorch DirectML (AMD Radeon GPU `privateuseone:0`)
*   Optimizer: AdamW (Learning Rate: 2e-5, Weight Decay: 0.01)
*   Epochs: 1
*   Batch Sizes: 8 (Train), 16 (Validation/Test)
*   Data Splits: 
    *   Train: 70,197 samples
    *   Validation: 13,488 samples
    *   Test: 13,113 samples

4. NOVEL MATHEMATICAL LOSS FUNCTIONS
----------------------------------------------------------------------
*   Triage Boundary Loss: A custom margin-based objective ($m = 0.5$) that calculates the distance between the logit of the true class and the maximum logit of the incorrect classes. It enforces a strict confidence gap to prevent routing hesitation.
*   KB Proximity Signal: A structural penalty that enforces a semantic distance between `REJECT` queries and the core domain embeddings. If a `REJECT` query embeds within a cosine-similarity threshold ($\tau_0 = 0.8$) of the KB, a Softplus loss penalty is applied.

5. EMPIRICAL RESULTS
----------------------------------------------------------------------
*   Test Loss: 0.4771
*   Validation Accuracy: 99.90%
*   Test Accuracy: 99.76%
*   Test TBP (Triage Boundary Precision): 99.76%
